## Why

rec53 已经具备较完整的功能面，但当前文档入口偏重、受众混合，代码中也存在少量启动/关闭边界和可读性问题，不利于以“可初步部署”的姿态发布 v1.0.0。发布前需要一次以收敛为目标的整理，明确默认推荐路径、减少文档噪音，并优先修复影响稳定性和维护成本的问题。

## What Changes

- 重组文档体系，明确区分用户文档与开发者文档，README 仅保留项目定位、快速部署入口和文档导航。
- 新增面向使用者的部署、配置、运维、排障文档，突出最小可用配置与默认推荐路径。
- 新增面向开发者的架构、测试、代码约束、发布流程文档，降低后续维护与协作成本。
- 精简并校正文档中的过时内容，消除 README、架构文档和零散说明之间的重复与职责混淆。
- 对启动、就绪等待、优雅关闭、配置校验等关键路径做小范围稳定性优化，并补充针对性测试。
- 对核心热路径代码做有限度的可读性整理，减少重复分支和模糊注释，避免过度设计和大规模重构。
- 为 v1.0.0 准备发布基线，明确“默认推荐”“可选增强”“实验能力”的边界。

## Capabilities

### New Capabilities
- `user-docs`: 提供面向使用者的部署、配置、运维与排障文档体系。
- `developer-docs`: 提供面向开发者的架构、开发、测试与发布文档体系。
- `release-readiness`: 提供 v1.0.0 发布前的默认路径校验与收敛要求。

### Modified Capabilities
- `readme-slim`: README 从“混合型总文档”调整为“首页与导航入口”，并增加用户/开发者文档分流。
- `architecture-doc`: 架构文档改为明确服务开发者，删除与用户部署指南重复的内容，并修正文档与代码结构不一致的问题。
- `startup-robustness`: 启动失败、就绪等待和后台任务初始化需要更稳健的失败路径与诊断行为。
- `dns-server-startup`: 服务启动与关闭流程需要补充对 listener 就绪和后台 goroutine 生命周期的约束。

## Impact

- 主要影响文件：`README.md`、`README.zh.md`、`docs/architecture.md`、新增 `docs/user/*` 与 `docs/dev/*`。
- 主要影响代码：`cmd/rec53.go`、`server/server.go` 及相关测试文件。
- 无新增外部依赖，重点是文档重组、代码收敛和测试补强。
- 对外行为以兼容为主；若有文档入口或推荐路径调整，将在 changelog/发布说明中明确说明。

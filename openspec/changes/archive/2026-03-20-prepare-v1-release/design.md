## Context

rec53 当前已经具备面向 v1 的主体能力，但文档职责边界不清晰：README 同时承载项目介绍、部署说明、架构说明和特性细节，`docs/architecture.md` 也混入了使用者视角的内容。代码层面整体可用，但 `server` 启动/关闭路径存在进一步收敛空间，尤其是 listener 就绪等待、后台 goroutine 生命周期和失败诊断这类与“可部署”直接相关的边界。

这次变更是一次发布前收敛，而不是新一轮功能建设。约束如下：

- 不新增外部依赖，不改变项目部署模型。
- 不做大规模状态机重构，不重写缓存、IP 池或 XDP 设计。
- 优先确保默认路径清晰可部署，其次再处理高级能力的文档定位。

## Goals / Non-Goals

**Goals:**

- 建立清晰的文档分层：README 只做入口，用户文档与开发者文档分离。
- 为用户提供明确的最小部署路径、配置入口和排障路径。
- 为开发者提供持续维护所需的架构、测试、约束和发布文档。
- 收敛启动/关闭关键路径中的稳定性问题，并以测试锁定行为。
- 用小范围代码整理提高可读性，避免继续积累发布前技术债。

**Non-Goals:**

- 不引入新的核心解析能力。
- 不重新设计状态机抽象或模块边界。
- 不承诺在 v1.0.0 前完成大规模 benchmark、性能重构或 XDP 功能扩展。
- 不把所有现有说明都翻译成双语；优先保证双语 README 和中文开发文档可用。

## Decisions

### 1. 文档按受众拆分，而不是按原文件继续堆叠

决策：

- `README.md` / `README.zh.md` 仅保留项目定位、快速开始、最小配置和文档索引。
- 新增 `docs/user/`，承载部署、配置、运维和排障。
- 新增 `docs/dev/`，承载架构、开发、测试、代码约束和发布流程。

原因：

- v1.0.0 的第一需求是“能部署、能理解、能维护”，不是“所有信息都放首页”。
- 同一文档面向两类受众会持续膨胀，最终同时伤害首屏体验和维护体验。

备选方案：

- 保持现状，仅继续精简 README。
  放弃原因：只能缓解首页冗长，不能解决使用者和开发者内容混杂的问题。

### 2. 架构文档面向开发者，而不是面向所有读者

决策：

- `docs/architecture.md` 继续保留为核心开发者文档，并由 `docs/dev/architecture.md` 或索引文档指向它。
- 文档内容只保留模块职责、请求流程、状态机、缓存和并发边界，不再重复部署与配置操作说明。

原因：

- 架构文档应该解释“系统如何工作”，不应承担“用户如何运行”的职责。
- 这样能减少 README 与架构文档之间的重复和漂移。

备选方案：

- 继续在 `docs/architecture.md` 中同时写部署与实现。
  放弃原因：受众混杂，长期一定再次失控。

### 3. 代码优化只做关键路径收敛，不做结构性重写

决策：

- 优先处理 `cmd/rec53.go`、`server/server.go` 及相关测试中的启动、就绪等待、关闭、配置校验问题。
- 仅在必要时提取小函数、压缩重复分支、澄清注释。

原因：

- v1 发布前最大的风险不是“写得不够新”，而是“默认路径边界不够稳”。
- 小步收敛更容易测试和回归验证。

备选方案：

- 对状态机和 server 生命周期做系统性重构。
  放弃原因：回归面太大，不符合发布前节奏。

### 4. 为功能做分级表达

决策：

- 在文档和发布准备中明确三层能力：
  - 默认推荐：递归解析、hosts、forwarding、缓存、warmup、metrics
  - 可选增强：snapshot、pprof、SO_REUSEPORT
  - 平台/实验特性：XDP

原因：

- 可以降低使用者对高级特性的认知负担，也能避免把平台特定能力误读成默认依赖。

备选方案：

- 所有特性同等展示。
  放弃原因：会弱化默认部署路径，提升首次部署复杂度。

## Risks / Trade-offs

- [文档拆分后链接漂移] → 通过 README 索引和开发者入口页集中收敛链接，避免“多入口失联”。
- [精简过度导致信息遗漏] → 将详细内容下沉到 `docs/user/*` 和 `docs/dev/*`，README 只删重复，不删必需信息。
- [启动/关闭路径修改引入回归] → 仅修改边界控制逻辑，并用包级测试覆盖成功路径和失败路径。
- [v1 范围继续膨胀] → 在 tasks 中明确“文档收敛 + 稳定性修复 + 发布基线”，不引入新能力开发。

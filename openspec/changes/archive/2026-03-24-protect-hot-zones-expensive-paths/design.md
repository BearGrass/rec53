## Context

rec53 现有请求路径已经形成清晰分层：`HOSTS_LOOKUP`、`FORWARD_LOOKUP`、`CACHE_LOOKUP` 负责本地廉价路径，只有在 forwarding 外部查询或 `CACHE_LOOKUP_MISS` 之后进入 iterative / upstream 时，才真正消耗昂贵解析资源。`v1.3.1` 已经在 requester 维度完成了 per-client 昂贵路径保护，但它只能回答“谁占用了昂贵路径”，还不能回答“多个 requester 是否共同把同一个热点名字空间持续推入昂贵路径”。

这次 change 的业务目标不是构建通用 anti-DDoS 系统，也不是直接保护上游出包资源；它要解决的是单热点 zone 的入口保护：当多个 requester 在短时间内持续把同一个 zone 的新请求推入昂贵路径时，rec53 应在入口 admission 层隔离这类热点，避免它拖累其他 zone 的正常解析。

当前讨论已经明确几条约束：

- 保护范围只在收包后的昂贵路径入口，不扩展到上游出包并发控制
- 昂贵路径定义沿用 `v1.3.1`：forwarding 外部查询、`CACHE_LOOKUP_MISS` 后 iterative / upstream
- 只阻止新的昂贵路径进入，不影响 `hosts` hit、forwarding hit、cache hit 等 cheap path
- 常态记录先按 coarse business-root key 做轻量占用记录，优先级为 forwarding zone、基础后缀之上的第一层业务域、最后才回退到三级域名；二级公共后缀层如 `com.`、`net.` 不纳入保护候选
- 一次只保护一个热点 zone
- 热点需要连续多个短窗口成立，主要是给正常业务瞬时启动 / 初始化流量留缓冲
- observe mode 只在全局昂贵占用接近机器承受上限时触发，而不是在轻量偏离平时安全区时触发
- 第一版阈值使用内部常量，不在本 change 中扩成用户可调参数面

## Goals / Non-Goals

**Goals:**
- 为多个 requester 共同打热同一个 zone 的场景增加入口保护，而不是只保护单个 requester。
- 保持与 `v1.3.1` 一致的昂贵路径语义，避免形成第二套“何为昂贵请求”的定义。
- 让系统能够在保持常态记录轻量的前提下，从短时昂贵流量中动态选出当前最需要保护的单个热点 zone，而不是依赖静态手工配置热点名单。
- 在热点 zone 进入保护后，只拒绝新的昂贵路径进入，尽可能保留 cheap path 的可服务能力。
- 保持 operator 语义清晰：这是对“热点名字空间持续制造昂贵成本”的临时保护，而不是永久封禁域名。

**Non-Goals:**
- 不处理多个 zone 同时升温造成的全局性压力；那属于后续 global / upstream 保护版本。
- 不扩展到上游出包并发保护、authoritative/back-end 保护或系统总 QPS 保险丝。
- 不做长期攻击信誉、自动黑名单、模式识别或机器学习判定。
- 不把 cheap path 一并封禁，也不把受保护 zone 变成完整域名黑洞。
- 不在第一版同时实现多热点并行保护、优先级队列或热点排行榜型控制面。

## Decisions

### 只保护单热点 zone，而不是处理所有多 IP 昂贵流量

第一版将问题收敛为“多个 requester 共同把同一个 zone 推入昂贵路径”的场景。多 IP 打很多不同 zone 导致的总体压力，故意不由本变更处理，而是留给后续 upstream/global 保护版本。

这样做的原因：
- 把 zone protection 与 global protection 的职责明确分开
- 让 operator 能用“热点隔离”而不是“全局限流”来理解这个版本
- 防止一开始把不同层级的问题混成同一套 admission 语义

备选方案：
- 直接把所有多 IP 昂贵流量都纳入同一个保护模型，会让 zone、upstream、global 三种问题边界模糊，难以解释也难以验证。

### 常态记录绑定 coarse business-root key，三级域名只是 fallback

第一版的常态轻量记录绑定到 coarse business-root key，而不是永远绑定到字面三级域名。优先级是：命中的 forwarding zone、基础后缀之上的第一层业务域、最后才回退到三级域名。以当前约定的层级口径看：`.` 为一级、`com.` 为二级、`example.com.` 为三级，因此 `example.com.` 是最小公网 fallback 业务层级，而 `com.`、`net.` 等公共后缀层不进入保护候选。进入 observe mode 后，系统只围绕当前热点 coarse key 做单层下钻，寻找是否存在更具体的保护对象。

这样做的原因：
- 常态记录保持极简，避免让低概率保护事件影响正常业务
- coarse business-root key 能统一 forwarding、基础后缀与 fallback 三级域名三种来源，避免文档和实现出现两套 coarse key 语义
- 三级域名 fallback 通常已经具有明确业务或品牌语义
- 公共后缀层过宽，保护它们没有明确业务价值，且容易误伤过多正常流量
- observe mode 下再做单层下钻，可以兼顾实现简单与保护对象精度
- 若攻击者通过大量不同 zone 共同制造压力，应该由后续出包保护或全局保险丝处理，而不是由 zone protection 承担

备选方案：
- 常态直接对所有层级做 suffix 树分析会把观察成本带入正常业务热路径。
- 强制从四级及以下开始会错失 `example.com.` 这种已经足够具体、且具有业务意义的热点父域。

### 业务根优先使用 forwarding zone，其次基础后缀，再退回三级域名

zone key 的最小业务根来源按以下优先级决定：

- 若请求命中 forwarding zone，则优先使用该 forwarding zone 作为业务根
- 否则匹配“默认值 + 追加值”构成的基础后缀集合，并从基础后缀之上的第一层业务域开始归类
- 若前两者都未命中，则回退到三级域名

这样做的原因：
- forwarding zone 是 operator 显式声明的业务边界，应优先于自动推导规则
- 默认值 + 追加值可以覆盖公网、K8s、常见云环境与企业私有环境，不要求软件理解全世界所有命名体系
- 三级域名 fallback 让系统在配置不完整时仍有保守兜底，而不是完全失去 zone 归因能力

这里的“默认值”只表示默认基础后缀集合的内容可参考或复用 warmup 已有的公共后缀思路，而不是让 zone protection 在语义上依赖 warmup 配置本身。内容可以复用，但能力定义和配置语义保持独立。

第一版建议默认基础后缀集合直接与当前 warmup 默认后缀保持一致，也就是复用 warmup 当前 curated TLD 集合作为默认基础后缀：`com.`、`cn.`、`de.`、`net.`、`org.`、`uk.`、`ru.`、`nl.`、`br.`、`xyz.`、`info.`、`top.`、`it.`、`fr.`、`au.`、`in.`、`us.`、`pl.`、`ir.`、`eu.`、`es.`、`ca.`、`io.`、`ai.`、`me.`、`site.`、`shop.`、`online.`、`biz.`、`app.`。

这样做的原因：
- 直接对齐 warmup 默认后缀，可以避免第一版同时维护两套不同的默认集合
- 这让 operator 对“默认基础后缀从哪来”有更稳定的心智，也降低实现与文档分叉风险
- 私有环境差异继续通过 append 机制处理，保持第一版默认值足够小、足够稳
- 如果部署环境没有补充私有后缀，系统仍有 forwarding zone 优先级和三级域名 fallback 可作为保守兜底

备选方案：
- 只靠固定根层级会在 K8s / 私有域环境中失去业务意义。
- 只靠环境配置而没有 fallback，会让漏配场景直接失去 zone protection 能力。

### 使用“最小充分热点后缀”选择单个保护对象

系统不直接保护最宽父域，也不直接保护最深叶子，而是在 observe mode 中围绕热点三级域名选择“最小充分热点后缀”。第一版业务规则是：

- 只做单层下钻，不做递归持续下钻
- 如果某个 child 占据父域聚合昂贵 occupancy 的至少 `80%`，则保护 child
- 如果没有 child 占据主要部分，则保护父域
- 如果某个 child 占据父域 `100%` 的观察流量，则直接选该 child 并停止继续分析
- 一次只选一个当前最突出的热点 zone

这样做的原因：
- 可以区分“整个父域都热”和“某个更深子树明显异常”
- 单层下钻已经能显著降低误伤整个父域的风险，同时避免递归下钻把保护对象追到过细叶子
- 能在存在正常兄弟流量的背景下，把异常热点从父域中剥出来
- 一次只保护一个对象，行为更容易解释；若压力持续，系统自然会在后续窗口切换到下一个热点

备选方案：
- 直接保护最宽父域过于粗暴，会把很多正常子树一起卷入保护。
- 递归持续下钻虽然更精细，但会增加观察复杂度，并更容易把保护对象追到过细层级。
- 同时保护多个热点虽然更激进，但第一版会显著增加状态和解释复杂度。

### sibling 判断看占比，不看倍数

child 是否值得继续下钻，第一版用“占父域昂贵流量的主要部分”来判断，而不以 sibling 倍数差作为核心标准。

这样做的原因：
- sibling 可能本来只有 1-2 次请求，倍数会非常不稳定
- 占比更贴近“如果保护更深 child，是否已经拦住这波昂贵流量的大头”这个真实问题
- 在存在正常业务背景流量的前提下，占比更稳健地表达了“异常热点是否相对兄弟明显集中”

备选方案：
- 用倍数判断会在兄弟样本很小的时候产生大量噪声，过于敏感。

### observe mode 与热点确认使用简单窗口聚合，而不是单窗口瞬时值

第一版的热点识别与单层下钻统一基于最近几个短窗口内 occupancy-time 的简单求和，不引入加权、平滑或动态衰减。热点选择不看单窗口瞬时值，而看最近几个窗口内的聚合结果。

这样做的原因：
- 能降低单窗口抖动对热点判断的影响
- 与“给业务瞬时启动留缓冲”的目标保持一致
- 简单求和最容易解释、测试与回放验证

备选方案：
- 对最近窗口赋更高权重虽然更灵敏，但会增加解释和测试复杂度。

### 热点需要连续多个短窗口确认后才进入保护

zone protection 不是单窗口立即出手的门神，而是带缓冲的热点隔离器。候选热点必须在多个连续短窗口内都成立，系统才将其置为受保护状态。第一版使用 `5s` 短窗口，并要求同一个候选在连续 `3` 个观察窗口内成立后才进入保护。

这样做的原因：
- 主要是给正常业务的瞬时启动、初始化或短促突发留出缓冲
- 减少因为单个短窗口尖峰而误伤正常业务
- 使“受保护状态”更接近持续热点，而不是短暂 burst

备选方案：
- 单窗口立即保护虽然更激进，但更容易把正常业务瞬时启动误判为攻击性热点。

### observe mode 由平均昂贵并发接近核数上限触发，整机 CPU 作为 guardrail

observe mode 不是因为系统“比平时忙一些”就触发，而是因为系统已经接近机器承受边界才触发。第一版主指标采用短窗口内的平均昂贵并发数：

- `avg_expensive_concurrency = 总 occupancy-time / 窗口时长`
- 第一版短窗口为 `5s`
- 最近 `3` 个短窗口的 occupancy-time 简单求和用于 observe mode 下的热点选择和下钻判断
- 当 `avg_expensive_concurrency >= 0.75 * runtime.NumCPU()` 时，认为昂贵并发已接近核数上限
- 整机总体 CPU 利用率 `>= 70%` 作为高压 guardrail，与主指标一起决定是否进入 observe mode

这样做的原因：
- occupancy-time 比请求数更接近系统真实资源占用
- 归一化为平均昂贵并发后，更容易与机器并发能力做比较，也更容易向 operator 解释
- 整机 CPU 作为 guardrail 能降低“昂贵占用高但系统仍有大量余量”时的误触发，并与“保护 OS 可承受并发边界”的产品目标保持一致

备选方案：
- 仅仅偏离平时安全区就触发 observe mode 会让该能力过于敏感，背离“低频启用”的目标。
- 只看 CPU 会丢失对 resolver 自身昂贵路径占用的业务归因。

### 进入保护后，只拒绝新的昂贵路径进入，并继续复用 `REFUSED`

受保护状态不是对某个 zone 做完整封禁。系统只拒绝该 zone 的新昂贵请求继续进入 forwarding 外部查询或 iterative / upstream 路径；仍可由 `hosts`、forwarding hit、cache hit 等廉价路径完成的请求继续放行。第一版继续沿用 `REFUSED` 作为快速拒绝语义。

这样做的原因：
- 保护真正昂贵的资源，而不是粗暴封杀所有流量
- 减少误伤成本，保留已有 cheap path 能力
- 与 `v1.3.1` 一致地把 admission control 聚焦在昂贵工作，而不是整个请求生命周期
- 与 `v1.3.1` 保持一致的返回语义，有利于 operator 理解和排障

备选方案：
- 直接封禁整个 zone 虽然简单，但会把 cheap path 一并牺牲，误伤面过大。

### 保护退出使用 observe 触发前正常基线附近的简单回落规则

系统需要在进入 observe mode 前保留一个正常基线视图，而不是把“触发当下已经偏高的值”误当作退出基线。第一版保护退出不采用复杂曲线或分档力度，而是使用简单回落规则：当当前全局昂贵占用回落到 observe mode 触发前的正常基线附近时，即可退出保护。第一版安全余量固定为 `1.05x`，也就是 `current <= pre-trigger-baseline * 1.05` 时退出保护。

这样做的原因：
- 回落到正常基线附近就退出，符合“尽量少启用”的目标
- 用 observe 触发前的少量窗口构造基线，仍然足够简单，不需要维护复杂历史模型
- 能避免保护状态长时间滞留，减少对正常业务的额外时延影响

备选方案：
- 抛物线式或连续强度释放虽然更细腻，但第一版没有必要引入保护强度曲线模型。

## Risks / Trade-offs

- [常态记录影响正常业务] -> 平时只做 coarse business-root key 的轻量 occupancy 记录；更复杂的下钻和热点判断仅在 observe mode 下触发。
- [热点选择过宽] -> 通过“默认停在父域、只有强证据才下钻 child”的高门槛规则，降低误把整个父域过早纳入保护的风险。
- [热点选择过细] -> 第一版只做单层下钻，不递归追到更深叶子；child 占父域 `100%` 时直接停止下钻，但不把这个停止条件抬升为比主规则更重要的核心机制。
- [瞬时业务 burst 被误判] -> 候选必须连续多个短窗口成立后才进入保护，给初始化/启动流量留缓冲。
- [K8s / 私有域层级与公网域名规则不一致] -> 采用“forwarding zone > 基础后缀 > 三级域名 fallback”的业务根选择顺序，并允许管理员追加环境后缀。
- [默认基础后缀与 warmup 语义耦合] -> 只复用 warmup 默认公共后缀的内容集合，不复用其配置语义或生命周期语义。
- [observe mode 触发过于敏感] -> 仅在平均昂贵并发接近 `NumCPU()` 上限且总体 CPU 进入高压区时才触发观察。
- [zone 保护与 global/upstream 保护语义混淆] -> 在 proposal/design 中明确本变更只覆盖“单热点 zone 的入口保护”，不处理多 zone 全局性压力。
- [只保护一个热点会漏掉其他热点] -> 第一版接受这种保守性，以换取更简单的行为；若压力持续，系统将在后续窗口自然切换或重新选出新的热点。
- [operator 难以理解为何某个 zone 被保护] -> 后续 spec 必须要求聚合指标与日志能够解释：当前保护对象、窗口内热点选择依据，以及 cheap path 未受影响的边界。

## Migration Plan

第一版预期以默认关闭或仅在明确配置下启用的方式引入，避免在没有 operator 心智准备时直接改变热点 zone 的请求行为。上线顺序建议为：

1. 先补齐热点选择与受保护状态的聚合观测
2. 在压测和开发环境验证热点识别与误伤边界
3. 再决定真实拒绝模式的默认配置与 rollout 节奏

若上线后发现热点识别过宽或对正常启动流量误伤过大，应可以直接关闭该能力，回退到仅保留 `v1.3.1` 的 per-client 保护。

## Implementation Notes

- 当前实现继续把能力挂在既有 `ExpensiveRequestLimitModeEnabled` 之下：未启用昂贵路径保护时，不额外引入热点 zone 入口保护。
- 第一版只在 forwarding 外部查询入口与 `CACHE_LOOKUP_MISS` 后的 iterative 入口执行热点 zone admission check；不扩展到上游出包阶段，也不处理 multi-zone 全局压力。
- 默认基础后缀直接复用 `DefaultCuratedTLDs`，并支持 operator 通过 `dns.hot_zone_base_suffixes` 追加私有环境 suffix。
- 当前 CPU guardrail 通过 Linux `/proc/stat` 采样整机利用率实现；若平台不可提供等价信息，则热点 zone observe mode 会保持保守。

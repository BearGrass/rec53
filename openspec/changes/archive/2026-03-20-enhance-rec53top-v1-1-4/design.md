## Context

当前 `rec53top` 的 detail 内容已经具备诊断价值，但交互仍然停留在 MVP 阶段：

1. overview 没有持续可见的“当前焦点”，用户需要记住 `1-6` 才能进入 detail。
2. detail 虽然能用 `0` / `Esc` 返回，但当前路径更像“记热键”，而不是“可探索的终端界面”。
3. footer help 仍然主要列出按键，没有把“你现在正选中哪个面板、下一步能做什么”表达得足够清楚。

`v1.1` 还没有结束，因此下一步不应切到运行韧性，而应先补齐 TUI 最基本的导航完成度。

## Goals / Non-Goals

**Goals**

- 在 overview 中引入明确焦点，并允许通过方向键、`j/k/l`、`Tab`/`Shift-Tab` 导航。
- 允许用 `Enter` 从当前焦点直接进入对应 detail。
- 在 footer/help 中表达当前焦点和主要操作，降低“必须背热键”的门槛。
- 保持现有 `1-6`、`0`、`Esc`、`q`、`r`、`h/?` 兼容。

**Non-Goals**

- 不在这一版实现多层 drill-down 树或 panel 内二级页面。
- 不改变 detail 信息结构本身。
- 不做历史 sparkline、排序筛选或多 target。

## Decisions

### 1. 使用“overview 焦点”而不是立即引入多层 drill-down

- overview 维持 2x3 六面板布局。
- 始终有一个当前焦点面板。
- 焦点通过边框标题或视觉标识突出。
- `Enter` 进入焦点 detail，`Esc`/`0` 返回 overview 并保留焦点。

原因：

- 这是最小但最有体感的交互增强。
- 不需要重做数据模型，也不会引入多层导航复杂度。

### 2. 保留数字键为快速入口，但把它降级为快捷方式

- `1-6` 继续直接进入对应 detail。
- 新用户默认可以只依赖方向键 / `j/k/l` + `Enter`。

原因：

- 兼容现有使用方式。
- 让 TUI 更接近“可发现”的终端界面。

补充说明：

- `h` 已经是现有 help 兼容键，因此 overview 导航不复用 `h` 做左移。
- 左移由方向键和 `Shift-Tab` 覆盖，避免破坏既有帮助入口。

### 3. footer 需要体现“当前焦点 + 主要动作”

- 简洁 help 下显示主要键位和当前焦点。
- 展开 help 时补充导航键和状态说明。

原因：

- 当前帮助文本偏静态，不够贴近当前交互状态。

## Risks / Trade-offs

- 新焦点状态会给渲染逻辑增加少量状态管理复杂度。
  通过固定 6 个 panel 的索引映射来控制，避免引入通用组件框架。

- overview 焦点样式如果过重，会影响原有状态色阅读。
  焦点提示只增强边框/标题，不覆盖面板内容状态颜色。

- 键位增加后可能和现有输入捕获冲突。
  保持现有键位优先级，新增键位仅补充 overview 导航。

## Migration Plan

1. 为 overview 六面板建立固定索引和相邻导航规则。
2. 更新输入捕获逻辑，加入方向键、`j/k/l`、`Tab`、`Shift-Tab`、`Enter`，并保留 `h` 为 help。
3. 为焦点面板增加清晰但克制的视觉提示。
4. 更新 footer/help 与用户文档。
5. 补充 TUI 导航测试并验证旧键位仍兼容。

## Context

rec53 在 `v1.1.1` 后已经具备比较完整的 Prometheus 指标面，文档里也已经定义了 dashboard 布局和排查顺序。但当前这套能力默认假设使用者愿意先部署 Prometheus 或 Grafana，或者至少会手工写 PromQL。对于 rec53 当前强调的单机、node-local、开发机自测场景，这一步仍然过重。

我们需要一个本地终端看板，把已有 `/metric` 数据直接转成“打开就能看”的运维界面，同时继续保持当前版本线的小步推进原则。约束如下：

- 不把 TUI 代码混入 `server/` 或主服务进程。
- 不要求 Prometheus server、Grafana 或其他外部组件在线。
- 不修改现有 metrics 名称、标签或暴露路径来迁就 TUI。
- 首版必须明显小于“完整监控平台”，否则会挤占后续 `v1.1.3` 运行韧性工作。

## Goals / Non-Goals

**Goals:**

- 提供一个单独的本地终端命令，默认读取 `http://127.0.0.1:9999/metric`。
- 将 `traffic / cache / snapshot / upstream / XDP / state machine` 六类信号收敛为固定布局面板。
- 在没有 Prometheus 的情况下，也能从连续抓取两次以上的数据中计算基础速率、比例和退化状态。
- 对目标不可达、指标缺失、XDP 未启用等情况提供明确降级显示，而不是空白或崩溃。
- 明确首版边界，使 `v1.1.2` 可以作为一个可实现、可验证、可发布的独立版本。

**Non-Goals:**

- 不支持多 target 聚合、远程集群列表或跨节点对比。
- 不做持久化历史趋势、导出报表、告警规则管理。
- 不把 TUI 直接包装进 `rec53` 主进程。
- 不为了 TUI 重命名或重构现有 Prometheus 指标体系。
- 不在首版实现 detail drill-down、面板排序/筛选、sparklines 或交互式历史视图。

## Decisions

### 1. 以独立命令 + 独立目录实现，而不是嵌入主服务或塞进 `rec53ctl`

决策：

- 新增独立命令，暂定命名为 `rec53top`。
- TUI 主要实现放到单独目录，例如 `tui/`，命令入口只负责参数和生命周期。
- `rec53ctl` 首版不集成 TUI 包装命令。

原因：

- 这样最符合“单独开目录、不要混进主服务路径”的要求。
- 不修改 `rec53ctl` 可以减少 bash 脚本侧分支和发布耦合，先把 TUI 本体做稳。
- 独立命令的失败、依赖和刷新行为不会影响主 DNS 进程。

备选方案：

- 把 TUI 直接做进 `rec53ctl top`。
  放弃原因：会同时拉入脚本接口设计、安装分发和二进制定位问题，超出 MVP。

- 把 TUI 嵌进 `rec53` 主服务。
  放弃原因：主服务是守护进程，不适合附带终端 UI 生命周期。

### 2. 直接抓取 `/metric` 文本并在本地计算派生值，不依赖 Prometheus API

决策：

- TUI 直接 HTTP GET 指标端点，解析 Prometheus text exposition。
- 只消费 MVP 需要的 metric family，并维护上一次采样快照，在本地计算 rate、ratio 和状态摘要。

原因：

- 这样符合“无需先部署 Prometheus”的业务目标。
- 当前 metrics 面已经稳定，直接消费 `/metric` 是最短路径。
- 本地保存最近一次或最近几次快照即可支撑 2s-5s 刷新频率下的基础速率展示。

备选方案：

- 依赖 Prometheus server 的 query API。
  放弃原因：把零依赖自测路径重新变成了依赖外部组件。

- 只展示绝对值，不做任何速率或比例计算。
  放弃原因：会明显削弱“是否正在退化”的判断价值。

### 3. 固定六块面板，首版只读，不做自由布局和复杂交互

决策：

- 首版布局固定覆盖：traffic、cache、snapshot、upstream、XDP、state machine。
- 只支持基础交互：退出、手动刷新、帮助、可选 target 覆盖。
- 默认展示当前值、短窗口速率和关键状态摘要，不做自由增删 panel。

原因：

- 固定布局能直接复用 `v1.1.1` 已经定义好的 dashboard 阅读顺序。
- 对 rec53 这种单机工具，首版最有价值的是“快读”，不是“可配置到像 Grafana”。

备选方案：

- 允许用户自定义 panel 布局和 PromQL。
  放弃原因：这本质上在向监控平台演进，范围失控。

### 4. 对缺失和不可达状态显式建模，优先保证解释性

决策：

- 目标不可达时显示 `disconnected` / `stale` 状态和最近错误。
- 指标族缺失时显示 `unavailable`，不把其误判为零值。
- XDP 相关指标缺失且服务正常时，按 `disabled or unsupported` 处理。

原因：

- 对运维来说，“没有数据”和“指标为 0”是完全不同的语义。
- rec53 当前很多部署不会启用 XDP，TUI 必须把这个状态解释清楚。

备选方案：

- 统一按 0 展示缺失项。
  放弃原因：会制造错误健康判断。

### 5. 接受一个终端 UI 依赖，但把 TUI 复杂度限制在只读 dashboard

决策：

- 引入成熟的 Go TUI 依赖来处理刷新、按键、布局和窗口尺寸变化。
- 依赖只服务于 TUI 命令，不反向侵入 `server/`、`monitor/` 或 `cmd/rec53.go`。

原因：

- 手写 ANSI 刷新会把精力浪费在终端兼容细节，而不是面板表达。
- 用成熟库更容易做出可读、稳定、稍微“像样”的界面。

备选方案：

- 完全手写 ANSI 输出。
  放弃原因：实现成本不一定更低，且后续维护更脆弱。

### 6. 首版以 `top` 式稳定工作台为基底，但保留后续增强入口

决策：

- 首版采用“顶部 summary + 固定分域面板 + 底部最小键位”的工作台布局，而不是纯对象列表式 `top` 或重交互式资源浏览器。
- 首版在内部把 scrape、view-model、panel render 分层，给后续增强留入口，但不在 `v1.1.2` 中实现这些增强功能。
- 明确把以下能力列为后续可扩展项：detail 视图、面板 drill-down、历史 sparklines、排序/筛选、更多快捷键。

原因：

- rec53 的核心对象是少量异构问题域，而不是大量同构资源行，纯 `top` 布局并不贴合。
- 直接做重交互会显著抬高 MVP 风险，但如果不预留分层，后续扩展时又容易推倒重来。

备选方案：

- 完全按 `top` 风格做成单列表。
  放弃原因：难以自然承载 `cache / snapshot / upstream / XDP / state machine` 这种异构健康域。

- 首版就做出 detail/趋势/排序能力。
  放弃原因：超出 MVP，容易把实现重心从“可读、可用”拉向“可玩、可配”。

## Risks / Trade-offs

- [终端 UI 依赖引入额外维护面] → 依赖只用于独立命令，保持主服务零影响。
- [Prometheus 文本解析与指标演进脱节] → 只读取已在 `docs/metrics*.md` 明确约定的稳定指标，并用测试锁住关键映射。
- [速率计算因采样过短而噪声偏大] → 首版明确标注为短窗口速率，并允许手动刷新与固定刷新间隔。
- [面板过多导致终端过于拥挤] → 采用固定摘要视图，优先放“是否正常、哪里退化”，而不是塞进所有明细。
- [版本范围再次膨胀] → 明确把多 target、历史趋势、交互式 drill-down 推迟到后续版本或 backlog。
- [首版实现把后续增强路堵死] → 从一开始就分离 scrape、状态聚合和 render 层，避免未来只能重写。

## Migration Plan

1. 先冻结 MVP 规格：独立命令、单 target、固定六面板、只读、直接抓 `/metric`。
2. 新增 TUI 目录与入口命令，打通 scrape、parse、view-model、render 基础链路。
3. 以假的 metrics snapshot 和最小集成测试锁住关键面板输出与降级状态。
4. 更新 README、用户文档和 roadmap，提供本地自测步骤。
5. 发布时按“本地运维体验增强”说明，不将其描述为新的监控平台。

回滚策略：

- 如果 TUI 依赖或实现不稳定，可以整体回退该独立命令与文档，不影响主服务与现有 metrics。

## Open Questions

- 命令名最终是 `rec53top`、`rec53-tui`，还是其他更贴合项目命名的形式。
- 刷新频率默认取 `2s` 还是 `5s` 更平衡噪声与可读性。
- 首版是否需要在页脚暴露最近一次 scrape 时间和抓取耗时。

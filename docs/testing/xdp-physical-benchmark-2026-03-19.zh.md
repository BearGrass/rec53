# 物理直连 XDP 多域名压测报告（2026-03-19）

## 测试目的

验证 `rec53` 在两台物理主机通过网线直连、开启 XDP 快速路径后的多域名缓存命中性能，并给出一份可复现的用户侧测试记录。

## 测试环境

### 拓扑

- 服务端：本机，`192.168.53.1/24`
- 客户端：远端主机，`192.168.53.2/24`
- 连接方式：两台机器通过网线直连
- 协商速率：`1000Mb/s`、全双工、自动协商开启

### 服务端

- CPU：11th Gen Intel Core i7-1165G7（4C8T）
- 内核：`6.8.0-101-generic`
- 直连接口：`enp89s0`
- 网卡驱动：`igc`
- rec53 监听：`192.168.53.1:53`
- metrics 监听：`192.168.53.1:9999`
- XDP：开启，挂载接口为 `enp89s0`
- 运行配置：

```yaml
dns:
  listen: "192.168.53.1:53"
  metric: "192.168.53.1:9999"
  log_level: "info"
  listeners: 1
warmup:
  enabled: false
xdp:
  enabled: true
  interface: "enp89s0"
```

### 客户端

- CPU：AMD Ryzen 7 4800H（8C16T）
- 内核：`6.17.0-19-generic`
- 直连接口：`eno1`
- 网卡驱动：`r8169`
- 压测工具：仓库内置 `tools/dnsperf`

### 查询集

使用仓库自带的 [`tools/dnsperf/queries-sample.txt`](../tools/dnsperf/queries-sample.txt)。

- 共 `13` 个查询项
- 混合了 `A / AAAA / MX / TXT / NS`
- 代表的是“多域名、以缓存命中为主”的压测场景，不是单域名极限冲榜

## 测试方法

### 预热

正式压测前，先在客户端做一轮小并发预热：

```bash
./tools/dnsperf/dnsperf -server 192.168.53.1:53 -f /tmp/query-multi.txt -c 4 -d 15s
```

预热结果：

- Queries: `320,708`
- QPS: `21,374.4`
- Timeouts: `0`
- Errors: `0`
- P50: `159us`
- P95: `288us`
- P99: `370us`

### 正式压测

客户端同时启动 `4` 个 `dnsperf` 进程，每个进程并发 `50`，持续 `180s`：

```bash
for i in 1 2 3 4; do
  ./tools/dnsperf/dnsperf -server 192.168.53.1:53 \
    -f /tmp/query-multi.txt -c 50 -d 180s > /tmp/dnsperf-multi-$i.out &
done
wait
```

总并发约为 `200`。

## 结果解读

### 1. 这轮结果是稳定且可信的

- 压测时长拉到 `180s` 后，结果明显比之前的 `5s`、`15s`、`30s` 更稳定
- 全程 `0 timeout`、`0 error`
- 4 个 `dnsperf` 进程之间的 QPS 很接近，没有明显抖动

### 2. 多域名场景下，当前平台可以稳定在约 32 万 QPS

这轮结果可以把当前环境的一个用户可参考数字写成：

`rec53 + XDP + 两台物理主机直连 + 13 域名缓存命中 + 总并发 200 + 180 秒`

约等于：

`320K QPS`

### 3. 当前瓶颈更像是测试平台，而不是 rec53 没有生效

虽然服务端 XDP 已经工作正常，但客户端网卡是 `r8169`，并且实测只有单个接收中断向量，`receive-hashing` 也是关闭且不可调的。这个条件下，回包收包能力很容易先成为瓶颈，所以当前数字更像是“这套测试平台的上限”，不一定是 `rec53` 的最终上限。

### 4. 这个数字不等于千兆线速极限

如果拿“纯 UDP 包转发”或“更强网卡、更好多队列、更理想压测器”的预期去比，`320K QPS` 不是极限值。  
但对当前这套环境来说，它已经说明：

- 链路已经正常工作在千兆
- XDP 的命中路径已经真实生效
- 结果不是短时波动，而是 3 分钟稳定值

## 已知边界

- 这是缓存命中主导场景，不代表冷缓存或递归查询场景
- 查询集只有 `13` 个条目，不代表高随机域名压力
- 客户端使用 `r8169` 单队列能力较弱，限制了继续向上扩展
- 本次没有同步跑一组“同条件、同 180 秒、关闭 XDP”的对照组，所以这个报告重点是“XDP 已生效时的稳定值”，不是严格的 A/B 对照实验

## 建议的下一步

如果还要继续往上探上限，优先级建议如下：

1. 更换客户端网卡，优先 Intel 多队列网卡。
2. 设计多源流量，让回包能更好地分散到多个 RX queue。
3. 在同样 `180s` 条件下补一组 no-XDP 对照，做更严格的 A/B。
4. 如果目标是极限值而不是用户报告，可以进一步结合 IRQ/RSS/NUMA/绑核来做专项调优。

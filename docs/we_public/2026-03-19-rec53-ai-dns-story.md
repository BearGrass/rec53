# rec53：一个递归 DNS 的继续施工，以及我为什么把 XDP 也一起做了

很久以前就想自研一个递归DNS，解决一些性能问题，后来写了一半刚把功能调通就耽搁了，现在AI编程出来了，狠狠的续写了一把，效果比我想象中的好多了。

这篇不打算写成“项目感想”，而是按技术分享的方式，把这个项目现在到底做到了什么、为什么这么设计、以及这次 benchmark 到底测出了什么，系统地讲一遍。

项目叫 `rec53`。  
它是一个用 Go 写的递归 DNS，核心关注点不是“功能堆得多全”，而是下面这几件事：

- 递归路径要清晰
- 缓存命中路径要短
- 可观测性要足够好
- 要能继续往 XDP/eBPF 方向推进

如果只用一句话概括：

> `rec53` 不是一个“能解析就行”的 DNS demo，而是一个围绕性能路径持续打磨的递归 DNS 工程。

---

## 一、为什么还要自己写一个递归 DNS

递归 DNS 这个领域并不缺项目，但如果你真的关心性能和行为可控性，会很快发现一个问题：

很多现成实现能用，但不一定容易深入优化。

更具体一点，我真正在意的从来不是“配置项有多少”，而是这些问题：

- 缓存命中路径到底有多短
- 查询状态机是不是显式的
- 热路径上还有多少不必要的分配
- 故障发生时，能不能快速定位在哪个状态退化
- 用户态缓存命中已经够快了吗
- 有没有必要继续把命中路径前移到内核

这些问题一旦成立，你就很自然会想自己把递归 DNS 这条链路重新捋一遍。

所以 `rec53` 的起点并不是“我想重新发明轮子”，而是：

> 我想有一个自己完全看得懂、完全能下手改、还能拿来做性能实验的递归 DNS。

---

## 二、项目定位

如果把 `rec53` 当成一个技术项目来介绍，它当前的定位大概是这样：

- 一个偏性能导向的递归 DNS
- 以缓存命中路径优化为核心
- 有清晰的递归状态机
- 有基本完整的观测面
- 支持 XDP/eBPF 缓存快速路径实验

它适合什么场景：

- 自建递归 DNS 的技术验证
- 分析缓存路径和递归路径
- 做 XDP 在应用协议场景下的实验
- 作为一个便于理解和继续重构的 DNS 基础工程

它当前不强调什么：

- 不是“企业级一体化 DNS 套件”
- 不是“插件系统优先”的平台
- 不是“所有生产特性都已经补齐”的成熟替代品

这个定位其实很重要，因为它决定了后面的设计取舍。

## 三、架构上，rec53 在做什么

如果先不看 XDP，只看普通路径，`rec53` 的主干很简单：

```text
query
  |
  v
+----------------+
| DNS listener   |
+----------------+
  |
  v
+----------------+
| state machine  |
+----------------+
  |
  +------> cache hit ------> return response
  |
  +------> cache miss -----> recursive query -----> cache writeback -----> return
```

如果开启 XDP，就会多出一条更靠前的缓存命中路径：

```text
client packet
     |
     v
+----------------------+
| NIC / XDP hook       |
| parse DNS question   |
| lookup BPF map       |
+----------------------+
     |                           |
     | hit                       | miss
     v                           v
+------------------+      +------------------+
| XDP_TX reply     |      | Go DNS listener  |
| kernel fast path |      +------------------+
+------------------+               |
                                   v
                         +----------------------+
                         | recursive state      |
                         | machine              |
                         | init -> cache ->     |
                         | glue -> iter -> ret  |
                         +----------------------+
                                   |
                                   v
                         +----------------------+
                         | Go cache / warmup /  |
                         | snapshot / metrics   |
                         +----------------------+
                                   |
                                   +---- sync ----> BPF cache map
```

这个图里最关键的不是“盒子画得好不好看”，而是它表达了一个明确的观点：

- 缓存命中尽量留在最前面处理
- 缓存未命中再进入 Go 的递归状态机
- Go 缓存写回时继续同步给 XDP 使用

也就是说，`rec53` 并不是把 XDP 当成一套平行系统，而是把它放在“已有递归 DNS 工程”前面，作为一条面向 cache hit 的加速层。

---

## 四、递归状态机为什么要拆开

递归 DNS 的一个常见问题是：逻辑虽然能跑，但路径揉在一起之后，很难判断某个请求到底在哪个阶段变慢、回退或者失败。

所以 `rec53` 这边比较明确的一件事，就是把递归过程拆成可推导的状态。

可以把它理解成下面这条主链：

```text
STATE_INIT
   |
   v
IN_CACHE ?
   | yes
   +-----------------------------> RET_RESP
   |
   | no
   v
CHECK_RESP / IN_GLUE / IN_GLUE_CACHE
   |
   v
ITER
   |
   v
RET_RESP
```

拆状态机的收益很直接：

- 每个状态更容易单测
- 每个分支更容易打日志
- 行为回归更容易发现
- 优化可以更准确地落在某个状态上

在写递归 DNS 这种系统时，我越来越觉得：

> 状态机显式化，本身就是一种性能工程准备工作。

因为很多性能问题和正确性问题，最后都不是“整个系统慢”，而是“某个状态在某类域名上退化了”。

---

## 五、缓存为什么是第一优先级

递归 DNS 日常最有价值的优化，其实不是“递归查得更快”，而是“尽量别走递归”。

所以 `rec53` 里，缓存不是一个附属能力，而是整个数据面的中心。

当前已经具备的相关能力包括：

- DNS 缓存
- warmup 预热
- snapshot 快照
- XDP map 同步

这个关系可以用一张小图说清楚：

```text
recursive response
                 |
                 v
         +------------------+
         | Go DNS cache     |
         +------------------+
            |          |
            |          +------> warmup / snapshot
            |
            +------> serialize + sync
                           |
                           v
                    +--------------+
                    | BPF cache    |
                    | map for XDP  |
                    +--------------+
```

这条链路的意义在于：

- Go 路径负责完整功能
- XDP 路径负责命中加速
- 两者不是割裂的，而是通过 cache sync 连接起来

## 六、为什么会继续做到 XDP

如果只是普通的 Go 用户态缓存命中，其实已经能跑到不错的数字。  
但 DNS 这个协议有个非常适合继续往下压路径的特点：

- 请求包很短
- qname / qtype 提取逻辑明确
- cache hit 场景可以很高频
- 返回响应的模式很固定

这就让它成为很适合做 XDP 的协议类型。

传统用户态命中路径大概是这样：

```text
NIC
 -> kernel network stack
 -> socket recv
 -> userspace handler
 -> Go cache lookup
 -> build response
 -> socket send
 -> kernel network stack
 -> NIC
```

如果走 XDP fast path，理想路径则变成：

```text
NIC
 -> XDP parse DNS
 -> BPF map lookup
 -> build response in-kernel
 -> XDP_TX
```

这个差异为什么值得做，不是因为“听起来高级”，而是因为它直接减少了：

- 用户态切换
- 协议栈往返
- goroutine 调度参与
- Go 热路径压力

当然，复杂度也会上来：

- BPF map 结构设计
- Go cache 和 BPF map 的同步一致性
- verifier 约束
- 驱动差异
- benchmark 真实性校验

所以这件事不是“把 XDP 加进去”这么简单，而是你得先有一个结构够清楚的 DNS 工程，才配去做它。

---

## 七、可观测性是必须先补齐的

做这类优化，最怕的不是“没优化成功”，而是“你以为优化成功了”。

所以 `rec53` 这次补的一个重点，是把关键观测面拉起来。

特别是 XDP 这部分，如果没有指标，最容易出现两种误判：

1. 你以为请求走了 XDP，实际上全部回退到 Go 路径。
2. 你以为性能问题在服务端，实际上可能是客户端、网卡或者压测方式的问题。

所以项目里专门暴露了这些指标：

```text
rec53_xdp_status
rec53_xdp_cache_hits_total
rec53_xdp_cache_misses_total
rec53_xdp_pass_total
rec53_xdp_errors_total
```

再加上启动日志、运行时诊断日志，基本能把下面这些问题看清楚：

- XDP 到底挂没挂上
- 命中是不是在涨
- miss 是不是异常偏高
- 有没有 error
- 包是不是大量走了 pass

对于这种系统，很多时候不是“多一个指标更完整”，而是：

> 没有这些指标，你根本没法认真谈 benchmark。

## 八、这次 benchmark 是怎么做的

这次我故意没有只拿短时间、单域名、理想环境的数字来写，而是做了一轮更像样的长时压测。

### 1. 测试环境

```text
server
------------------------------------------------
CPU      : Intel i7-1165G7 (4C8T)
kernel   : 6.8.0-101-generic
NIC      : enp89s0 / igc
address  : 192.168.53.1
XDP      : enabled

client
------------------------------------------------
CPU      : AMD Ryzen 7 4800H (8C16T)
kernel   : 6.17.0-19-generic
NIC      : eno1 / r8169
address  : 192.168.53.2
tool     : dnsperf
```

### 2. 网络拓扑

```text
+-------------------+         direct cable         +-------------------+
| client            | <--------------------------> | server            |
| 192.168.53.2      |                              | 192.168.53.1      |
| dnsperf           |                              | rec53 + XDP       |
+-------------------+                              +-------------------+

link mode: 1000Mb/s, full duplex
```

### 3. 工作负载

```text
query set
------------------------------------------------
13 domains / mixed qtypes
A / AAAA / MX / TXT / NS

load shape
------------------------------------------------
4 x dnsperf process
each process concurrency = 50
total concurrency ~= 200
duration = 180s

warmup
------------------------------------------------
small warmup run before benchmark
```

这里特意用了多域名和 180 秒时长，原因很简单：

- 单域名太容易测成“理想命中路径”
- 5 秒或者 10 秒太容易把短期波动当成真实结果
- 时间拉长以后，很多“好看但不稳”的数字会自己掉下来

## 九、benchmark 结果

这轮 3 分钟多域名压测的结果如下：

```text

==================================================
QPS        : 320,592.5
queries    : 57,722,268
timeouts   : 0
errors     : 0
P50        : 302us - 306us
P99        : 5.1ms - 5.3ms
duration   : 180s
```

再把它排成一张更适合公众号代码块的卡片，就是这样：

```text
+--------------------------------------------------+
| rec53 physical benchmark                         |
+--------------------------------------------------+
| topology   : 2 hosts / direct cable / 1GbE       |
| workload   : 13 domains / ~200 concurrency       |
| duration   : 180s                                |
| xdp        : enabled                             |
+--------------------------------------------------+
| QPS        : 320,592.5                           |
| queries    : 57,722,268                          |
| timeouts   : 0                                   |
| errors     : 0                                   |
| P50        : 302us - 306us                       |
| P99        : 5.1ms - 5.3ms                       |
+--------------------------------------------------+
```

从“benchmark 的可信度”角度来说，我更看重的是这几个点：

- 持续 180 秒，不是瞬时冲高
- 多域名，不是单域名刷 cache hit
- 物理机直连，不是 loopback
- 0 timeout，0 error

这个结果至少说明：  
当前这套实现已经不是“能跑的实验品”，而是进入了一个可以认真记录数据、认真讨论瓶颈的位置。

## 十、怎么确认它真的走了 XDP

这部分是整篇里最关键的一个问题。

因为如果不能确认包真的走了 XDP，那所有性能讨论都不成立。

这轮测试前后读取到的 XDP 指标增量大致是：

```text
xdp metrics delta
----------------------------------------------------
status        : 1
hit delta     : +57,757,820
miss delta    : +27,295
pass delta    : +8
error delta   : 0
hit ratio     : 99.95%
----------------------------------------------------
```

这个数字说明了几件事：

1. `rec53_xdp_status = 1`  
   说明 XDP fast path 处于工作状态。

2. `hit delta` 在大量增长  
   说明这轮压测里的请求并不是都掉回 Go 路径了。

3. `error delta = 0`  
   说明这轮物理网卡测试里没有出现 XDP 处理错误。

4. `hit ratio ~= 99.95%`  
   说明这轮 workload 确实是 cache-hit dominated，而且 fast path 真正在承压。

也就是说，这不是“看起来像在测 XDP”，而是可以比较明确地说：

> 这轮 3 分钟多域名压测，绝大多数请求都走了 XDP fast path。

## 十一、这些结果应该怎么解读

我觉得有三个结论是比较稳的。

### 1. XDP 在 rec53 里已经不是概念功能

很多项目的 XDP 支持停留在“能 attach”这一步。  
而 `rec53` 这边已经把这些链路串起来了：

- XDP attach
- BPF map lookup
- Go cache -> BPF map sync
- 命中 / 未命中统计
- 真实物理链路 benchmark

这已经不是 PPT 层面的“支持 XDP”，而是一个可测、可验证、可继续优化的工程实现。

### 2. benchmark 终于不是“拍脑袋数字”

这次不是 loopback，也不是短时峰值，而是：

- 物理机
- 实际网卡
- 真实链路
- 多域名
- 长时压测
- 指标反证

它当然不代表覆盖一切场景，但至少已经足够支撑一篇认真一点的技术分享。

### 3. 当前上限仍然更像平台瓶颈，不一定是项目极限

这次排查过程中也暴露了一个现实问题：

客户端网卡是 `r8169`，而且收包能力、队列能力相对弱，已经很可能先一步卡住整体上限。

所以当前这个 `320K QPS` 更接近的是：

> 在当前硬件和链路条件下，rec53 + XDP 的稳定实测值

而不是它的理论上限。

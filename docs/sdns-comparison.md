## 优先级判断标准

| 优先级 | 含义 |
|--------|------|
| **High** | 影响核心用户场景（生产部署、安全合规），应尽快补齐 |
| **Medium** | 增强竞争力，技术可行，值得规划 |
| **Low** | Nice-to-have，资源充足时可做 |
| **Out-of-scope** | 与 rec53 "轻量递归解析器"定位不符，明确不计划实现 |

## 9. 总结：rec53 的定位与差距

### rec53 的核心优势

rec53 在以下方面具备独特优势：

- **IPQualityV2 智能 IP 选择**：64 样本滑动窗口 + P50/P95/P99 + 4 态自动故障恢复，比 sdns 简单的 RTT 排序更精细
- **Happy Eyeballs 双路并发**：主 IP 和次优 IP 同时查询，取最快响应，降低尾延迟
- **NS 启动预热**：30 个精选 TLD 覆盖全球 85%+ 域名，有效消除冷启动延迟
- **清晰的状态机架构**：7 态流水线，便于审计和调试

### 高优先级差距（High）

这些功能对生产部署有直接影响，建议优先规划：

| 功能 | 影响 |
|------|------|
| DoT（DNS over TLS） | 生产环境加密传输标配，无此功能限制部署场景 |
| DoH（DNS over HTTPS） | 浏览器/移动端主流加密 DNS 协议 |
| DNSSEC 验证 | 安全合规要求，防止 DNS 劫持 |
| 缓存预取（Prefetch） | 消除热点记录的过期尾延迟 |

### 中优先级差距（Medium）

增强竞争力，技术可行：

QNAME 最小化、访问控制（IP ACL、速率限制、并发上限）、缓存清除 API、EDNS Cookie、Reflex 攻击防御、周期性根服务器 priming、缓存大小上限配置、hosts 文件支持

### 低优先级 / Out-of-scope

- **Low**：Dnstap、HTTP 管理 API、DNS 转发模式、插件系统、配置热重载、多出口 IP、DNAME 等——nice-to-have，暂不规划
- **Out-of-scope**：Kubernetes 集成——与 rec53 定位不符，有专用产品（CoreDNS）覆盖此场景

### 功能覆盖率对比

| 类别 | sdns 功能数 | rec53 已实现 | 覆盖率 |
|------|------------|-------------|--------|
| DNS 解析能力 | 20 | 12 | 60% |
| 传输协议 | 8 | 3 | 38% |
| 缓存系统 | 9 | 3 | 33% |
| 安全功能 | 9 | 2 | 22% |
| 访问控制 | 5 | 0 | 0% |
| 监控可观测性 | 14 | 6 | 43% |
| Kubernetes | 5 | 0 | 0%（定位不同） |
| 扩展架构 | 8 | 4 | 50% |

rec53 是一个**专注于高质量递归解析**的解析器，而非功能全集。若需完整的加密传输、K8s 集成或中间件生态，sdns 是更合适的选择；若需要轻量、可审计、具备精细 IP 质量感知的递归解析核心，rec53 是更好的起点。
